DevStride ships a built-in **Model Context Protocol (MCP) server** so you can connect AI assistants directly to your workspace. Once connected, your AI client can answer questions about your work, find and read items, and make changes on your behalf using natural language. For example, you can ask your assistant *"What items am I working on?"* and it will use DevStride's tools to look up your recent and assigned work.

::alert{type="info" title="Phase One"}
MCP support is in its first phase, and the set of available tools is actively expanding. Expect new capabilities to appear over time without any action on your part.
::

## Open the Connect AI page

In the left navigation sidebar, find the **Help** section and click **Connect AI (MCP)** (terminal icon). This opens a page titled **Connect your AI client (MCP)**.

This page is available to **all users** — it is not limited to administrators. Everything you set up here acts within the permissions you already have.

## The MCP Server URL

Your AI client connects to a single endpoint, shown on the page as a copyable **MCP Server URL**:

```
https://api.devstride.com/mcp
```

The server identifies itself to your client as `devstride-mcp`. You'll paste this URL (or the ready-made setup command for your client) into your AI assistant during setup.

## Choose an authentication method

The Connect AI page offers two ways to authenticate, switched with a toggle at the top of the panel:

- **OAuth 2.1** (recommended, and the default)
- **API Key**

Lead with **OAuth 2.1** unless you have a specific reason to use an API key. The table below summarizes the difference:

| | OAuth 2.1 | API Key |
| --- | --- | --- |
| Stores a long-lived secret | No — short-lived token, refreshed automatically | Yes — you must copy and store the secret |
| Works with web clients (Claude.ai, ChatGPT) | Yes | No |
| Works with Claude Code, Codex, Cursor, Copilot, Gemini CLI | Yes | Yes |
| Organization scope | One org, chosen when you sign in | One org, chosen when you create the key |

::alert{type="warning" title="A connection acts as you"}
Whichever method you choose, the connection invokes any tool **you** can perform in the DevStride UI — including destructive actions such as archiving boards and items, deleting comments or time entries, and bulk-updating items. There are **no read-only or no-destructive scopes yet**. Treat an MCP connection like a set of credentials with your full level of access, and consider connecting from a lower-privilege account where possible.
::

## Method 1: OAuth 2.1 (recommended)

OAuth 2.1 is the simplest and safest option for most people. On first use, your AI client opens a browser so you can sign in to DevStride, pick which organization to connect to, and approve access. No secrets are stored on your machine — your client receives a short-lived token that it refreshes automatically in the background.

The in-app callout describes it this way:

> On first use your client opens a browser to sign in to DevStride and pick an organization. No secrets are stored — access is a short-lived token the client refreshes automatically.

### How OAuth scoping works

- **One organization per connection.** When you sign in, the consent page presents an organization picker and an **Allow** button. The connection is bound to exactly the organization you choose. To work in a different organization, you authorize again and pick that org.
- **You must be an active member.** You can only authorize an organization where your membership is active.
- **Short-lived access.** Access tokens last about an hour and refresh automatically, so you rarely sign in more than once per client.

This is an authorization flow (it grants access to the API), not a single sign-on / identity login.

## Method 2: API Key

API keys are the alternative for clients that can use them. You can create a key in two places:

- From the Connect AI panel, using the **Create a new API key** link.
- From **My Account → API Keys**, using the **New Key** button.

In the create dialog, you provide:

- A **Label** to identify the key.
- A **For Organization** selection — keys are scoped to a single organization at creation, so the organization always travels with the key.

When you create the key, the dialog displays the **Api Key**, **Organization Id**, **Organization Slug**, and **Api Secret** — and the secret is shown **only once**.

::alert{type="danger" title="Copy your secret immediately"}
The dialog warns: **"COPY YOUR API SECRET NOW! It cannot be viewed or reproduced later."** The secret cannot be regenerated. If a key is ever leaked, revoke it and issue a new one.
::

After creation, the dialog conveniently embeds the Connect AI panel pre-filled with your new credentials (locked to the API Key method) so you get a ready-to-paste setup command.

::alert{type="note" title="Who can create API keys"}
Any user can create an API key for any organization they belong to — key creation is not restricted to administrators. Because a key carries your full permissions for that organization, a good practice is to issue keys against a low-privilege account rather than an owner or admin account.
::

### Headers used by API-key clients

When a desktop or CLI client connects over HTTP with an API key, it sends four headers:

- `X-DevStride-Api-Key`
- `X-DevStride-Api-Secret`
- `X-DevStride-Organization-Id`
- `X-DevStride-Organization-Slug`

The setup snippet generated for your client includes these automatically — you normally won't set them by hand.

## Set up your AI client

The panel is organized into a tab for each supported client, with a ready-to-paste snippet, a contextual hint, and (for some clients) setup screenshots. The client tabs are:

- **Claude (web)**
- **Claude Code**
- **ChatGPT**
- **Codex**
- **Cursor**
- **Copilot**
- **Gemini CLI**

Each client's snippet is tailored to how that client expects to be configured:

- **Claude Code** and **Codex** give you a terminal `mcp add` command.
- **Copilot** provides a `.vscode/mcp.json` block.
- **Cursor** provides a `~/.cursor/mcp.json` entry.
- **Gemini CLI** provides an `mcpServers` entry for `~/.gemini/settings.json`.
- **Claude (web)** and **ChatGPT** surface a connector **Name** (use `DevStride`), a **Description** (ChatGPT), and the **MCP Server URL** to paste into the client's connector settings.

::alert{type="info" title="Web clients use OAuth only"}
**Claude (web)** and **ChatGPT** can't send custom headers, so they only appear under the **OAuth 2.1** method. The in-app hint puts it plainly: *"Heads up: claude.ai and ChatGPT (web) can't send API-key headers — use the OAuth 2.1 method for those."* If you switch to the API Key method while you're on a web client tab, the panel falls back to the Claude Code tab.
::

### Connecting Claude (web)

For Claude.ai, add DevStride as a custom connector:

1. In Claude, open **Settings → Connectors** and choose **Add custom connector**.
2. For **Name**, enter `DevStride`.
3. Paste the **MCP Server URL**: `https://api.devstride.com/mcp`.
4. Click **Add**, then **Connect**, and sign in to DevStride when prompted.

::alert{type="note" title="Free-plan limit"}
On the Claude.ai free plan you can add only one custom connector.
::

### Connecting ChatGPT

ChatGPT also uses the OAuth method and the same connector **Name** and **MCP Server URL**, plus a connector **Description** (the panel provides a ready-to-paste one). Open **Settings → Connectors → Create**, paste the URL as the connector URL, and set authentication to OAuth. To get full access to DevStride's tools, turn on **Developer mode** under **Connectors → Advanced**.

### Reconnecting

If you need to reconnect Claude Code — for example, to switch organizations or refresh authorization — remove the existing server first:

```
claude mcp remove devstride
```

Then run the setup command again from the Connect AI panel.

## What your AI assistant can do

Once connected, your assistant can use a broad set of tools grouped by area of DevStride, including:

- **Workspace and identity** — confirm who you are, get workspace context, list teams and members, and surface your recent work.
- **Finding and reading items** — search items, open an item, and resolve names to boards, teams, users, labels, and folders.
- **Editing work** — create and update items, bulk-update, archive and unarchive, copy, reparent, move to a board or cycle, rank, mark done, and reopen.
- **Comments** — add, list, update, delete, reply, and react.
- **Boards, cycles, and time entries** — list and create boards, manage cycles and timeboxes, and add, start, stop, or check active time entries.
- **Relationships, the activity log, and quality checks** — add and remove relationships, review activity, and run quality checks on your items and boards.

A built-in help tool returns curated example prompts, so you can ask your assistant for ideas on how to get started.

The complete, authoritative list of tools is part of the MCP server itself and continues to grow as new capabilities ship.

### Working with sub-tasks (sub-items)

::alert{type="note" title="Terminology"}
What the DevStride UI calls a **sub-task** is called a **sub-item** in the API and MCP tools — the terms are interchangeable. Sub-tasks are lightweight rows with a simple status of **new**, **in progress**, or **done**, and they can't live on folders.
::

Your assistant can work with sub-tasks through dedicated tools — searching across sub-tasks, listing the sub-tasks of a parent, and getting, creating, updating, or deleting them. Item search also has an option to include items in your assignee results when one of their sub-tasks is assigned to you, so you can ask for "everything I'm working on, even via sub-tasks."

::alert{type="info" title="Permission-scoped results"}
Sub-task searches are filtered by the read permission of the parent item, so your AI client only ever sees sub-tasks under items you're allowed to read.
::
